# 参赛名称与开源代码、组件使用情况说明

## 1. 推荐参赛名称

项目参赛名称：**序刻清单 PlainList**

名称释义：

- “序刻”强调项目围绕时间序列展开，覆盖“当下、日、周、月、年”等多尺度信息组织方式。
- “清单”对应项目的核心能力，即任务、习惯与打卡事项的清单化管理。
- 保留 `PlainList` 作为英文标识，便于与当前代码仓库名称、项目内部标识和已有产品认知保持一致。

一句话定位：

**序刻清单 PlainList** 是一个面向个人习惯养成与任务规划的多尺度时间管理 Web 平台，集成终端式登录引导、日计划录入、周月年可视化追踪和主题插件扩展能力。

## 2. 当前项目自研实现范围说明

根据当前仓库结构与源码实现情况，项目下列内容由项目自身设计与开发完成：

- 前端应用壳层、终端式登录交互界面、日/周/月/年多视图面板。
- 计划管理、打卡记录、语言切换、主题预览与插件商店等前端业务逻辑。
- 后端鉴权、计划管理、打卡管理、插件管理等接口模块。
- 共享类型定义、数据校验结构、数据库迁移脚本与种子数据脚本。
- 页面布局、视觉表达、交互流程以及图表指标组织方式。

补充说明：

- 当前仓库未见独立引入的第三方业务源码目录。
- 当前项目未引入现成后台模板或大型 UI 组件库，界面主要以 Vue 单文件组件方式自主搭建。
- 当前项目所使用的开源依赖，主要承担前后端框架、图表渲染、数据校验、鉴权支持、构建测试等通用基础能力。

## 3. 开源代码与组件使用情况

以下统计依据当前仓库中的 `package.json` 文件整理，版本号以项目当前依赖声明为准。

### 3.1 前端框架与可视化组件

- `vue` `^3.5.29`
  用途：构建前端页面、组件系统和响应式交互逻辑。
- `pinia` `^3.0.4`
  用途：管理认证、计划、打卡、语言和插件等前端状态。
- `echarts` `^6.0.0`
  用途：渲染计划完成度、周度趋势、雷达图、柱状图等数据可视化内容。

说明：上述组件用于提供基础框架与图表渲染能力，具体页面结构、业务数据组织方式及交互逻辑由项目自行实现。

### 3.2 后端框架与基础能力组件

- `express` `^5.1.0`
  用途：构建后端 HTTP 服务及业务接口。
- `mysql2` `^3.15.2`
  用途：连接 MySQL 数据库并执行数据读写。
- `zod` `^4.1.11`
  用途：进行接口参数和共享数据结构校验。
- `bcryptjs` `^3.0.2`
  用途：进行用户密码加密处理。
- `jsonwebtoken` `^9.0.2`
  用途：生成与校验用户登录令牌。
- `cors` `^2.8.5`
  用途：处理前后端分离场景下的跨域访问。
- `dotenv` `^16.6.1`
  用途：加载环境变量配置。

说明：上述开源组件用于实现服务框架、数据库访问、鉴权与安全校验等通用能力，项目业务接口设计与数据模型组织为自主实现。

### 3.3 工程化、构建与测试工具

- `typescript` `^5.9.3`
  用途：统一前后端与共享包的 TypeScript 开发。
- `vite` `^7.3.1`
  用途：前端本地开发与生产构建。
- `@vitejs/plugin-vue` `^6.0.4`
  用途：为 Vite 提供 Vue 单文件组件支持。
- `vue-tsc` `^3.1.1`
  用途：进行 Vue 项目的类型检查与构建前校验。
- `tsx` `^4.20.6`
  用途：运行后端与数据库脚本的 TypeScript 开发命令。
- `vitest` `^3.2.4`
  用途：进行前后端单元测试。
- `eslint` `^8.57.1`
  用途：进行代码规范检查。
- `@typescript-eslint/eslint-plugin` `^8.44.1`
  用途：补充 TypeScript 代码检查规则。
- `@typescript-eslint/parser` `^8.44.1`
  用途：为 ESLint 解析 TypeScript 代码。
- `eslint-plugin-vue` `^9.33.0`
  用途：补充 Vue 组件检查规则。
- `vue-eslint-parser` `^9.4.3`
  用途：解析 Vue 单文件组件。
- `concurrently` `^9.2.1`
  用途：并行启动共享包、前端和后端开发服务。

补充说明：

- `@types/node`、`@types/cors`、`@types/express`、`@types/jsonwebtoken` 等依赖主要用于 TypeScript 类型声明支持。
- 工程化类开源工具主要服务于项目开发、调试、测试与构建流程，不直接构成项目业务功能本体。

## 4. 开源组件使用边界说明

为保证项目参赛说明清晰、合规，当前项目对开源代码与组件的使用边界可概括为：

- 项目以通用开源框架和基础库作为底层支撑，不依赖现成整站模板完成核心业务。
- 项目的页面结构、功能组织、交互流程、数据模型和业务逻辑由项目自主设计与实现。
- 图表、鉴权、状态管理、类型校验、数据库连接等基础能力依托成熟开源组件完成，以提高开发效率和系统稳定性。
- 当前项目未使用闭源收费组件作为核心功能依赖。

## 5. 合规说明

- 本项目所使用的第三方开源依赖均通过标准包管理方式引入。
- 项目在开发和展示过程中遵循相关开源组件的使用规则，具体许可证信息以各组件官方仓库或发布页面为准。
- 如后续用于公开发布、部署或成果展示，可进一步补充依赖清单及对应许可证信息，形成更完整的开源合规材料。
